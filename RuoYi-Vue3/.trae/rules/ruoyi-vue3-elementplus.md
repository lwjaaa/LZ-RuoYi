# 若依 Vue3 + Element Plus 企业级项目 Trae 全局规则文件

说明：本规则文件适配 RuoYi（若依）框架 Vue3 版本，基于 Element Plus、Vite、Pinia 技术栈，严格遵循若依官方企业级开发规范，可直接复制粘贴到 Trae AI 工具的「规则和技能」中，生成的代码无需大幅修改即可集成到项目，兼顾规范性、可维护性和开发效率，适配企业级后台管理系统开发场景。

## 一、项目基础信息（必选，Trae 自动识别匹配）

### 1.1 核心技术栈

- 前端框架：Vue 3.0+（强制使用 Composition API + \<script setup> 语法糖，禁止使用 Options API）
- UI 组件库：Element Plus（最新稳定版，禁止混用其他 UI 库，如需扩展组件需基于 Element Plus 二次封装）
- 构建工具：Vite（遵循若依官方 vite.config.js 配置，不随意修改别名、代理等核心配置）
- 状态管理：Pinia（严格遵循若依框架 Pinia 目录结构和使用规范，禁止使用 Vuex）
- 路由管理：Vue Router（遵循若依动态路由、权限路由配置规范）
- 请求库：基于 axios 封装的 @/utils/request.js（若依自带，所有接口请求必须复用该封装，禁止直接使用 axios 原生请求）
- 代码规范：ESLint + Prettier（遵循若依默认配置，禁止修改核心校验规则）
- 样式方案：Scoped CSS + Element Plus 主题变量（支持按需引入，禁止全局样式污染）

### 1.2 项目目录结构（严格遵循若依规范，Trae 生成文件需对应目录）

- src/api/：接口请求封装目录，按业务模块划分文件夹（如 system、user、dept、erp），单个模块一个 js/ts 文件，禁止跨模块混放接口
- src/views/：页面组件目录，按业务模块划分文件夹，单个页面一个 Vue 文件，页面内组件拆分至当前文件夹下的 components 子目录
- src/components/：公共组件目录，仅存放全局可复用组件（如分页、弹窗、表单模板），业务相关组件放对应页面目录下
- src/store/：Pinia 状态管理目录，按模块划分，单个模块一个 store 文件，仅存储全局共享状态，页面私有状态用 ref/reactive 管理
- src/utils/：工具函数目录，仅复用若依自带工具（如 auth、dateUtils、errorCode），新增工具需符合若依命名规范并放入该目录
- src/directive/：自定义指令目录，复用若依自带权限指令（v-hasPermi、v-hasRole），禁止随意新增自定义指令
- src/layout/：布局组件目录，禁止修改若依默认布局（三栏式：侧边栏+顶部导航+主内容区），如需调整需基于原有布局扩展
- src/assets/：静态资源目录，图片、样式、字体等按类型划分子目录，图片使用相对路径，禁止使用绝对路径

## 二、企业级代码规范（强制遵循，确保团队协作一致性）

### 2.1 命名规范

- 文件命名：采用 kebab-case 命名法（小写字母+连字符），如 user-list.vue、system-role.js，禁止使用 PascalCase 或 camelCase 命名页面/接口文件
- 组件命名：页面组件采用 PascalCase 命名法（首字母大写），如 UserList、RoleManage；公共组件统一前缀为 Re（如 ReTable、ReForm），确保组件名唯一
- 变量命名：采用 camelCase 命名法，语义化命名，禁止使用单字母（如 a、b、c）或无意义命名（如 data1、temp），布尔类型变量前缀加 is/has（如 isLoading、hasPermission）
- 函数命名：采用 camelCase 命名法，动词开头，语义化清晰（如 getList、handleAdd、deleteUser），禁止使用模糊命名（如 doSomething、handleClick）
- 接口命名：按“请求方式+业务场景”命名，如 listUser（查询用户列表）、addUser（新增用户）、getUser（查询用户详情），与后端接口名保持一致
- 常量命名：采用 UPPER_SNAKE_CASE 命名法（大写字母+下划线），如 PAGE_SIZE、STATUS_NORMAL，集中定义在对应模块的常量文件中

### 2.2 Vue 组件开发规范

- 组件结构：严格遵循「template → script setup → style scoped」顺序，禁止打乱顺序
- template 规范： &#x20;
  - 标签闭合：所有标签必须闭合，单标签（如 img、input）使用自闭合写法（\<img />）
  - 缩进规范：使用 2 个空格缩进，标签嵌套层次不超过 4 层，避免过度嵌套
  - 指令使用：v-model、v-for、v-if/v-else 等指令规范使用，v-for 必须搭配 :key，且 key 为唯一值（禁止使用 index 作为 key）
  - Element Plus 组件：所有表单、表格、弹窗、按钮等均优先使用 Element Plus 原生组件，样式与若依系统保持一致
- script setup 规范： &#x20;
  - 导入顺序：先导入 Vue 相关（如 ref、reactive），再导入 Element Plus 组件，然后导入接口、Pinia、工具函数，最后导入自定义组件，导入语句按字母顺序排列
  - 变量声明：ref/reactive 声明在脚本顶部，按“全局变量 → 页面状态 → 临时变量”顺序排列
  - 函数声明：按“生命周期钩子 → 接口请求函数 → 事件处理函数 → 工具函数”顺序排列，生命周期钩子遵循 Vue3 执行顺序
  - Props 定义：如需父子组件传值，Props 需指定类型、默认值，必要时添加校验，禁止使用 any 类型
  - emits 定义：组件触发的事件需在 emits 中声明，语义化命名，禁止隐式触发事件
- style 规范： &#x20;
  - 所有 style 标签必须添加 scoped 属性，避免样式污染全局
  - 样式命名：采用 BEM 命名规范（块-元素-修饰符），如 .user-list\_\_search-form--active
  - 主题适配：使用 Element Plus 主题变量（如 $--color-primary）

### 2.3 接口请求规范（核心，避免接口调用混乱）

- 接口封装：所有后端接口必须封装在 src/api/ 对应模块下，单个接口函数仅负责请求，不处理业务逻辑，返回 Promise 对象
- 请求方式：严格对应后端接口，查询类接口用 get 方法（参数用 params），新增/修改类接口用 post 方法（参数用 data），删除类接口用 delete 方法，禁止混用请求方式
- 请求拦截：复用若依 request.js 自带的请求拦截器（携带 token、请求 ID），禁止重新定义拦截器
- 响应处理：接口请求成功后，直接返回响应数据（response.data），异常处理（如 401 登录过期、403 权限不足）复用若依自带的响应拦截器，无需额外处理
- 加载状态：所有接口请求必须添加 loading 状态（使用 Element Plus ElLoading 或页面内 isLoading 变量），请求结束后关闭 loading，提升用户体验
- 异常提示：接口请求失败时，使用 Element Plus ElMessage 组件提示错误信息，错误信息优先使用后端返回的 msg，无返回时使用默认提示（“操作失败，请稍后重试”）

### 2.4 权限控制规范（贴合若依企业级权限需求）

- 按钮权限：所有操作按钮（新增、编辑、删除、导出等）必须添加 v-hasPermi 指令，权限标识与后端保持一致（如 v-hasPermi="\['system:user:add']"）
- 角色权限：如需按角色控制显示，使用 v-hasRole 指令（如 v-hasRole="\['admin']"），禁止手动判断角色
- 路由权限：页面路由配置遵循若依动态路由规范，无需在页面内额外判断路由权限，由 router/permission.js 统一控制
- 数据权限：查询接口需携带当前用户数据权限参数（如 deptId），确保数据隔离，符合企业级数据安全要求

## 三、Trae AI 生成规则（核心，确保生成代码可直接复用）

### 3.1 通用生成要求

- 所有生成的代码必须符合上述代码规范，无需人工修改即可集成到项目，避免生成冗余、不规范代码
- 生成页面时，自动包含若依页面通用结构：查询表单 + 数据表格 + 分页组件 + 操作按钮，布局与若依系统保持一致
- 生成组件时，优先复用若依公共组件（如 ReTable、ReForm），无对应公共组件时，基于 Element Plus 二次封装，确保组件可复用
- 生成代码时，禁止引入项目中未安装的依赖，禁止修改若依自带的配置文件（如 vite.config.js、request.js）
- 生成的代码需包含完整的注释，关键函数、变量、Props 需添加注释，注释清晰简洁，符合企业级代码可维护性要求

### 3.2 页面生成规则（高频场景）

- 查询表单：使用 ElForm 组件，标签宽度统一为 120px，表单项间距一致，查询按钮与重置按钮并排，重置按钮点击后清空表单并重新查询
- 数据表格：使用 ElTable 组件，表格宽度自适应，表头居中，内容左对齐，序号列固定，操作列固定在右侧，包含编辑、删除等常用操作
- 分页组件：使用 ElPagination 组件，分页参数（pageNum、pageSize）与若依后端接口保持一致，默认 pageSize 为 10，支持页码切换、每页条数切换
- 新增/编辑弹窗：使用 ElDialog 组件，弹窗标题区分“新增 XX”“编辑 XX”，表单校验与查询表单一致，提交按钮需添加 loading 状态，关闭弹窗后重置表单
- 删除确认：使用 ElMessageBox 组件，弹出确认提示（“确定要删除该数据吗？删除后不可恢复”），确认后执行删除接口，删除成功后刷新表格

### 3.3 组件生成规则

- 公共组件：生成的公共组件需具备通用性、可配置性，支持 Props 传值和 emits 事件触发，禁止绑定具体业务逻辑
- 业务组件：生成的业务组件需与当前业务模块对应，放入对应页面目录下的 components 子目录，仅服务于当前页面，避免全局复用
- 表单组件：生成的表单组件需包含完整的校验规则，校验规则与后端接口校验保持一致，支持必填、格式、长度等校验
- 表格组件：生成的表格组件需支持分页、排序、筛选、多选等功能，适配若依接口返回格式（{ rows: \[], total: 0 }）

### 3.4 避坑规则（禁止生成以下内容）

- 禁止生成 Options API 代码，所有代码必须使用 Composition API + \<script setup>
- 禁止生成 Vuex 相关代码，状态管理必须使用 Pinia
- 禁止生成直接使用 axios 原生请求的代码，必须复用 @/utils/request.js
- 禁止生成无 scoped 的 style 标签，避免样式污染
- 禁止生成冗余代码（如重复的工具函数、无用的变量），代码简洁高效
- 禁止生成与若依规范冲突的代码（如目录结构错误、命名不规范）

## 四、企业级扩展规范

- 导出功能：生成导出按钮时，复用若依自带的 exportExcel 工具函数，导出格式为 xlsx，文件名包含模块名称和导出时间
- 字典渲染：页面中需要渲染字典值（如状态、性别）时，使用若依自带的 dict 指令（v-dict），禁止手动渲染字典值
- 代码复用：相同业务逻辑（如表单校验、表格渲染）需封装成 hooks 或工具函数，避免重复编码
