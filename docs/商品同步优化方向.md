最优先风险


变体编辑后不会同步到 Shopify。

ShopifySyncExecutor.java (line 403) 里有 shopifyVariantId 就直接跳过，所以价格、SKU、规格图、库存策略、选项值改了都不会更新远端。Shopify 官方也明确 productUpdate 不支持更新 variants，应使用 productVariantsBulkUpdate： Shopify productUpdate。



本地删除变体不会删除 Shopify 变体。

ProductWizardServiceImpl.java (line 763) 只删本地变体和媒体，远端变体会残留。应补 productVariantsBulkDelete，官方 2026-04 有这个 mutation： productVariantsBulkDelete。



多店铺模型不成立。

erp_product.shopify_product_id 和 erp_product_variant.shopify_variant_id 是单字段唯一索引：20260324_erp.sql (line 39)、20260324_erp.sql (line 81)。如果一个 ERP 商品同步到多个 Shopify 店铺，会复用同一个 Shopify ID，状态也互相覆盖。SaaS 级应该是 product_id + store_id 维度维护商品、变体、媒体映射。



媒体同步是追加式，不是对账式。

productUpdate(..., media) 只是追加新媒体；本地删除媒体只删本地文件和记录：MediaServiceImpl.java (line 323)。远端旧图、排序、alt、规格图绑定不会被严格修正。



并发和幂等保护不足。

批量推送直接创建任务并异步执行：ProductServiceImpl.java (line 240)。没有 productId+storeId 锁、同步版本号、幂等键。两个任务同时推同一商品时，可能重复创建远端商品或互相覆盖状态。



Shopify 限流处理不够 SaaS 化。

当前没有读取 extensions.cost.throttleStatus，也没有按 250 input 限制拆批。Shopify 官方说明 GraphQL Admin API 按 cost 限流，数组输入最大 250： Shopify API limits。



预检主要在前端，后端同步入口可绕过。

前端有 readiness 检查，但 pushBatch 后端没有强制校验必填、图片、价格、SKU、店铺配置等：ProductController.java (line 124)。大厂级系统应在任务创建前做服务端 preflight，失败写任务明细，不调用 Shopify。



店铺详情接口有密钥泄露风险。

ShopifyStoreVo 包含 apiSecret 和 accessToken，getInfo 直接 objToVo 返回：ShopifyStoreVo.java (line 24)、ShopifyStoreController.java (line 79)。列表接口做了置空，详情接口没做。这个要优先修。



我建议的优化方向

先把同步改成“按店铺的商品对账模型”：本地商品是源数据，product_store_mapping / variant_store_mapping / media_store_mapping 记录每个店铺的远端 ID、状态、最后同步摘要。同步时按 diff 执行：商品主信息 update，变体 create/update/delete，媒体 create/delete/reorder，发布渠道单独控制。复杂商品可以考虑 Shopify 官方推荐的 productSet，它支持外部系统同步和 list 字段对账： productSet。

一句话：现在最大逻辑漏洞是“编辑后同步不等于远端一致”。先补 store-scoped 映射、变体/媒体 reconciliation、幂等锁和服务端预检，系统气质会立刻从“能推送”提升到“可运营、可追责、可规模化”。
